While Sundew can generate random personas automatically, custom personas let you deploy honeypots that mimic specific types of services. This is useful for targeted research, industry-specific deployments, or when you want precise control over what your honeypot looks like.

## Creating a Custom Persona

Create a YAML file with your persona definition:

```yaml
# my-persona.yaml
persona:
  company_name: "Quantix Trading Systems"
  industry: "fintech"
  api_style: "rest"
  framework_fingerprint: "spring"
  error_style: "structured"
  auth_scheme: "oauth2"
  data_theme: "trading"
  response_latency_ms: [50, 200]
  server_header: "Apache/2.4.57"
  endpoint_prefix: "/api/v3"
  seed: 173629
```

Start Sundew with your persona:

```bash
sundew serve --persona my-persona.yaml
```

## Field-by-Field Guide

### company_name

The fake company name that appears in API responses, documentation, and discovery files. Choose something that sounds plausible for the industry.

Good examples:
- "Northvane Analytics" (fintech)
- "Cedarpoint Health Systems" (healthcare)
- "Meridian Commerce" (e-commerce)
- "Arcline DevOps" (SaaS)

Avoid: names of real companies, obviously fake names ("Test Corp"), or anything that could cause confusion.

### industry

The overall theme that drives data generation. Supported values:

| Industry | Data theme | Typical endpoints |
|----------|-----------|-------------------|
| `fintech` | Transactions, accounts, balances | `/transactions`, `/accounts`, `/transfers` |
| `healthcare` | Patients, records, appointments | `/patients`, `/records`, `/appointments` |
| `saas` | Users, organizations, projects | `/users`, `/organizations`, `/projects` |
| `e-commerce` | Products, orders, customers | `/products`, `/orders`, `/cart` |

### api_style

The API architecture to mimic:

- `rest` -- Standard REST with JSON responses, the most common choice
- `graphql` -- GraphQL endpoint with schema introspection
- `grpc-web` -- gRPC-Web protocol over HTTP

### framework_fingerprint

Which web framework to mimic in HTTP headers and behavior:

| Value | Headers added | Error format |
|-------|--------------|--------------|
| `express` | `X-Powered-By: Express` | Express-style error pages |
| `django` | `X-Frame-Options: DENY`, `Vary: Accept, Cookie` | Django REST framework errors |
| `spring` | `X-Content-Type-Options: nosniff` | Spring Boot error responses |
| `rails` | `X-Runtime`, `X-Request-Id` | Rails-style error pages |

### error_style

How the API responds to invalid requests:

- `verbose` -- Detailed error messages with field-level validation info
- `terse` -- Minimal error codes
- `structured` -- RFC 7807 Problem Details format

### auth_scheme

The authentication mechanism to advertise and accept:

- `bearer_jwt` -- JWT tokens in Authorization header
- `api_key` -- API key in header or query parameter
- `basic` -- HTTP Basic authentication
- `oauth2` -- OAuth 2.0 with token endpoint

All schemes accept any credentials and return realistic tokens. The choice affects how the auth flow looks to visitors.

### data_theme

The type of fake data to generate. Usually matches the industry, but can be customized:

- `financial` -- Transactions, balances, account holders
- `medical` -- Patient records, diagnoses, appointments
- `users` -- User accounts, profiles, permissions
- `retail` -- Products, orders, inventory
- `trading` -- Market data, positions, trades

### response_latency_ms

A `[min, max]` range for artificial latency added to responses. This simulates real-world API performance and prevents timing-based fingerprinting.

Guidelines:
- `[50, 200]` -- Fast API (trading platform, real-time service)
- `[80, 350]` -- Standard API (typical SaaS)
- `[200, 800]` -- Slower API (healthcare system, legacy platform)
- `[500, 2000]` -- Deliberately slow (simulates heavy processing)

### server_header

The `Server` HTTP response header value:

- `nginx/1.24.0`
- `Apache/2.4.57`
- `gunicorn/21.2.0`
- `Microsoft-IIS/10.0`

Choose something consistent with the `framework_fingerprint`.

### endpoint_prefix

The URL prefix for all API routes:

- `/api/v2` -- Common modern API pattern
- `/v1` -- Simple versioning
- `/rest/v1` -- Explicit REST prefix
- `/api` -- No versioning

### seed

A deterministic random seed. Using the same seed with the same persona config produces identical response templates. Useful for reproducible research.

## Generating Response Templates

After creating your persona file, generate response templates:

```bash
# With LLM (recommended for unique content)
sundew generate --persona my-persona.yaml

# Preview what will be generated
sundew generate --persona my-persona.yaml --dry-run
```

The generated templates are saved to `./data/templates.json`. You can inspect and manually edit them if needed.

## Example: Targeted Deployment

Suppose you want to study how AI agents interact with healthcare APIs. Create a persona that mimics a hospital's patient management system:

```yaml
# healthcare-honeypot.yaml
persona:
  company_name: "Cedarpoint Health Systems"
  industry: "healthcare"
  api_style: "rest"
  framework_fingerprint: "django"
  error_style: "structured"
  auth_scheme: "oauth2"
  data_theme: "medical"
  response_latency_ms: [200, 800]
  server_header: "gunicorn/21.2.0"
  endpoint_prefix: "/rest/v1"
  seed: 582914
```

This deployment will:
- Serve endpoints like `/rest/v1/patients`, `/rest/v1/records`, `/rest/v1/appointments`
- Return Django-style headers and error responses
- Offer MCP tools like `search_patients` and `get_medical_record`
- Include fake patient data with realistic (but generated) medical records
- Use OAuth2 authentication flow

## Example: Multi-Persona Deployment

Deploy multiple Sundew instances with different personas to study how agents behave across varied targets:

```bash
# Terminal 1: Fintech honeypot on port 8080
sundew serve --persona fintech.yaml --port 8080

# Terminal 2: Healthcare honeypot on port 8081
sundew serve --persona healthcare.yaml --port 8081

# Terminal 3: SaaS honeypot on port 8082
sundew serve --persona saas.yaml --port 8082
```

Or use Docker Compose:

```yaml
services:
  fintech:
    image: sundewsh/sundew:latest
    ports: ["8080:8080"]
    volumes: ["./fintech.yaml:/app/sundew.yaml:ro", "./data/fintech:/app/data"]

  healthcare:
    image: sundewsh/sundew:latest
    ports: ["8081:8080"]
    volumes: ["./healthcare.yaml:/app/sundew.yaml:ro", "./data/healthcare:/app/data"]

  saas:
    image: sundewsh/sundew:latest
    ports: ["8082:8080"]
    volumes: ["./saas.yaml:/app/sundew.yaml:ro", "./data/saas:/app/data"]
```

## Validating Your Persona

After generation, verify your persona looks realistic:

```bash
# Start the honeypot
sundew serve --persona my-persona.yaml

# In another terminal, explore it
curl http://localhost:8080/.well-known/ai-plugin.json | jq .
curl http://localhost:8080/openapi.json | jq .
curl http://localhost:8080/api/v3/accounts | jq .
```

Check that:
- Company name and branding are consistent
- Endpoint paths match the prefix and industry
- Response data looks realistic for the theme
- Headers match the framework fingerprint
- Error responses match the error style
